## 2026-07-22 Adaptive Tiered Block Pipeline (Phase 2 of the Topics feature)

**Symptom:** N/A — new feature. Phase 1 gave every topic one flat gated quiz. The ask was to split robust topics (≥10 questions) into 2-6 sequential micro-blocks, each with its own formative check, followed by a cumulative Gate Check that unlocks the next topic; thin topics (<10 questions) get Self-Directed Instance-Lab instructions plus a single verification quiz instead. Worked out over an extended back-and-forth establishing hard constraints before any code was written (see the approved plan, same file path as the Phase 1 plan): `questions`/`options` permanently read-only, everything new derived or in a genuinely new table, replaces (not adds to) the single-quiz flow, and the existing global "Incorrect Answers Review" pool stays untouched — this pipeline gets its own topic-scoped local remediation instead.
**Root cause:** N/A — design decisions.
**Fix — data model, all additive:**
- `attempt_kind` ENUM gained exactly one new value, `'topic_block'` (`migration_14.sql`). `'topic'` deliberately kept its existing meaning ("the attempt that gates the unlock") — now specifically the Gate Check or thin-topic Final Verification — so every pre-existing pass-detection query (`topics.php`'s unlock chain, etc.) needed zero changes.
- `exam_attempts.block_number` (nullable) is the only new column. Block-to-question mapping is **never stored** — `csa_slice_block_questions()` derives which questions belong to block N by slicing the topic's ordered question-ID list at request time (remainder distributed to the earliest blocks, so a 23-question pool over 4 blocks is 6/6/6/5, not 6/6/6/5-with-a-gap). Current-block-for-a-user is likewise derived, not stored: `csa_compute_current_block()` mirrors `csa_compute_unlocked_topics()`'s exact chain-safe logic (a stale pass on block 3 without 1-2 doesn't skip you ahead).
- One new table, `topic_block_content` (registered in `table_prefix.php`'s `CSA_TABLES` — the third time this session a new table needed that step, and the second time a migration comment itself had to be reworded to avoid the same `csa_prefix_tables()` word-boundary rewrite that hit the topic-3 slug and a later migration comment — caught this time by writing a standalone test that runs the actual prefixing function against the migration file *before* moving on, instead of finding out after a staging deploy).
**Fix — backend:** Extracted `csa_start_topic_pipeline_attempt()` (`webapp/lib/topic_pipeline.php`) since block-quiz-start and Gate-Check-start share ~60 lines of identical DB orchestration (abandon stale attempt, fetch questions/options, shuffle option order, scale duration, insert, shape response) — only how the candidate question IDs get chosen differs between the two callers. Local remediation reuses the exact same `csa_build_topic_quiz_selection()` mixing function from Phase 1 unchanged; only the revision-pool SQL query's scope changed (`AND eat.topic_id = ?`, and for a block retry additionally `AND eat.block_number = ?`), replacing "missed anywhere across all topics" with "missed in this exact block" or "missed anywhere in this topic" for the cumulative Gate Check.
**Fix — frontend:** `topic-progress.js`'s view-model gained an `action` enum (`'block'|'gate'|'lab'|'locked'`) specifically so `topics.js` never has to re-derive the same branching logic by parsing the `ctaLabel` string — it just switches on `action` to decide which API to call.
**Verified live** (not just unit tests) with a throwaway local-only user: started Block 1 of an 11-question topic (correctly split 6/5), deliberately failed it 4/6, confirmed the retry surfaced exactly the 2 missed questions as revision picks, passed it, passed Block 2, confirmed the Gate Check only unlocked after both blocks passed, passed the Gate Check, confirmed the next topic unlocked — then confirmed a fresh topic's block quiz had zero revision-pool leakage from the first topic's misses (revisionCount: 0). Test user and its attempts deleted afterward. 107→118 PHP tests, 54→61 JS tests, all green throughout.
**Why this approach:** Badges and in-app social sharing were explicitly cut from this pass (deferred/dropped per direct instruction) rather than half-built — badge philosophy when it does happen is journey/experience-based, not a formal credential, and artwork is user-supplied. Lab-instruction content is drafted as baseline OOTB documentation, never marked `published`, until reviewed against the real instance — same content-accuracy rule Phase 1 established for lesson text, since there's no access to verify real ServiceNow menu paths against.

## 2026-07-22 Select-then-submit flashcards + category interleaving

**Symptom:** User feedback: the flashcard "click to flip" interaction (question → tap the card → answer appears) is passive review, not real self-testing — you never actually commit to an answer before seeing it revealed.
**Root cause:** N/A — deliberate redesign after a research-backed discussion, not a bug. The testing effect (Roediger & Karpicke and the replicated literature since) shows active retrieval beats passive review substantially for long-term retention; a flip-card without a selection step skips the retrieval step entirely.
**Fix:** Flashcard options are now real radio/checkbox rows (same `.option-row` markup and click-toggle/chooseN-cap logic as `exam.js`'s live exam-taking screen) that build up a `selectedLetters` array; a new "Submit Answer" button grades the selection and flips the card into a color-coded review — green for every correct option, red for a wrong pick, `(your answer)` tags — using the exact same CSS classes as the Mock Exam's post-submission review, so the two screens teach one consistent visual language. Grading itself is a new pure, tested module, `webapp/assets/js/lib/flashcard-grading.js` (`normalizeSelectedLetters()` / `isAnswerCorrect()`), which mirrors `webapp/lib/exam_grading.php`'s dedupe/sort/exact-set-match logic exactly rather than reimplementing it differently client-side — flashcards already receive each option's `correct` flag from `questions.php` (unlike the exam-taking endpoints, which withhold it pre-submission), so this needed no new API surface. 9 new JS tests in `tests/js/flashcard-grading.test.js`, mirroring `ExamGradingTest.php`'s cases 1:1. The existing Good/Again Leitner-box buttons stay fully manual, but one gets a `.suggested` glow (`box-shadow`, not disabled/forced) based on correctness — a nudge, since "got it right" and "how well I know this" aren't the same signal.
**Second change, same session:** `applyFilter()` now runs the resulting deck through an unbiased Fisher-Yates shuffle (`shuffled()` in `flashcards.js`) before rendering, so Study mode interleaves categories instead of presenting them in DB-insertion-order blocks. Interleaved (mixed-topic) practice is a well-replicated "desirable difficulty" (Bjork) that improves long-term retention and transfer versus blocked single-category practice, despite feeling harder in the moment; it also removes positional memorization ("3rd card in Forms is always B") as a substitute for real recall. Rapid Drill inherits the same shuffled `filtered` array for free; Match Game already had its own independent shuffle and is unaffected.
**Interaction fallout handled:** The swipe-to-reveal gesture (drag before answering) now calls `submitAnswer()` with whatever's currently selected — often nothing — rather than a separate "just reveal" path, so "I don't know, show me" naturally reuses the same grading code with an empty selection (documented and tested as "vacuously ungraded, never counted correct against a real answer key"). `#fcOptions`/`#fcSubmitBtn` were added to `cardEl`'s existing `pointerdown`/`click` exclusion lists — the same nested-button-inside-toggle-card bug class logged 2026-07-21 for "Show Me How"/"Add a note" would otherwise resurface here, since option rows and the new Submit button are freshly-added interactive descendants of `#flashcard`. Verified live in-browser (not just unit tests): checkbox rendering for a "Choose three" question, correct green/red coloring and `(your answer)` tags on a deliberately-wrong multi-select submission, the "Again" suggestion glow, Show Me How toggling without collapsing the card, click-to-restart on an answered card doing nothing before answering, a real Good-grade round-trip persisting to `flashcard_progress` via the network tab, and category interleaving (15 consecutive "All" cards spanned 11 different categories, no blocking). Suite: 81 PHP + 36 JS = 117.
**Added, not yet built (discussed but deliberately deferred):** an optional typed/free-recall mode (stronger retrieval per the generation effect, but ServiceNow answers are often full phrases — reliable fuzzy-matching is real scope, not a quick add) and explicit tracking of *which* wrong option gets picked repeatedly on a card (a specific-misconception signal distinct from the generic Leitner "needs review" bucket). Both flagged as v2 candidates, not started.

## 2026-07-21 Incorrect Answers Review learning loop (Mock Exam → Flashcards → Mini-Exam)

**Symptom:** No path existed from "you got a question wrong on the mock exam" back to targeted, verified-ready practice — a user who failed or scraped by had to either re-take the full 25-274 question exam or manually hunt down what they'd missed.
**Root cause:** N/A — new feature, not a bug.
**Fix:** Mini-exams are modeled as ordinary `exam_attempts` rows (`migration_11.sql`: `parent_attempt_id`, `attempt_kind ENUM('full','mini')`), reusing 100% of the existing grading/scoring/history code rather than a parallel system — the only new columns needed. `webapp/lib/incorrect_review.php` has two pure functions: `csa_compute_review_readiness()` (known-count / total ≥ 80% threshold) and `csa_pass_percent_for_kind()` (mini-exams need 80% to pass, full exams still 70% — read from `config['exam']['mini_pass_percent']`, falls back to 80 if a deployment's config predates this feature). Readiness signal reuses `flashcard_progress.status = 'known'` (Leitner box ≥ 2, i.e. two successful "Good" reviews) — the exact field that already means "mastered" everywhere else in the app — rather than inventing new tracking or a "Good/Easy" split that doesn't exist in the real flashcard UI (only Again/Good + a separate confidence slider). `exam_start.php` gained a `parentAttemptId` branch: server-derives the wrong-question set from `exam_answers WHERE attempt_id = parentAttemptId AND is_correct = 0` (never a client-supplied ID list, same rule as everywhere else scored data is involved), shuffles, and **caps at 15 questions** even if more were missed — mini-exams stay mini. `questions.php` gained an optional `?attemptId=` filter (same server-side derivation) so Flashcards can show only a specific attempt's misses. New `incorrect_review_status.php` endpoint returns the readiness computation for a given attempt. Frontend: `exam.js`'s results screen branches via a new pure `webapp/assets/js/lib/results-message.js` (`buildResultsMessage()` — headline + CTA struct, no DOM, fully unit tested) into three cases — full exam with misses → "Review Incorrect Answers with Flashcards" link; mini-exam pass → dashboard link; mini-exam fail → back into the same review loop. `flashcards.js` gained an `?mode=incorrect_review&attemptId=X` mode that fetches the filtered set, hides the normal filter/category/search/mode-switch UI (none of it applies to a fixed practice set), and shows a live readiness banner with a "Start Mini-Exam" button once the 80% threshold is hit.
**Design decision — flat, not chained, attempt linkage:** Every mini-exam's `parentAttemptId` is always the root full-exam attempt, never the previous mini-exam. This means (a) the flashcard review pool is always the *original* full set of misses, never narrowing round to round — matching the user's literal "same missed questions" spec — and (b) auditing "all mini-exam attempts for this full exam" is a single flat query instead of a recursive walk. The URL/CTA plumbing was simplified mid-session by dropping a redundant `originalAttemptId` param once this was settled — a single `attemptId` query param suffices everywhere since parent and current collapse to the same value.
**Verified live, not just unit-tested:** Full loop driven end-to-end through a real browser session against a local PHP+MariaDB stack (25-question exam → 6 wrong → flashcards review → 100% known → mini-exam pass with 80% mark shown; separately, a second 25-question exam with 25 wrong → capped to a 15-question mini-exam → deliberate fail → confirmed the "review again" link correctly routes back to the *root* attempt, and that the loop re-offers "Start Mini-Exam" afterward). This caught two real bugs the unit tests didn't: the mini-exam question count was originally unbounded (fixed with the 15-question cap above), and the local dev DB was missing `migration_10` entirely (unrelated pre-existing drift, applied to unblock testing).
**Why this approach:** User's original spec proposed 4 new denormalized columns (`incorrect_question_ids`, `mini_exam_passed`, `mini_exam_attempts`, `flashcards_reviewed`); countered with the 2-column `exam_attempts` extension above since every one of those 4 is either derivable from `exam_answers` (already the source of truth for correctness) or from reusing `exam_attempts`/`flashcard_progress` as real rows — avoiding a second, easily-divergent source of truth for data the app already tracks. User accepted the leaner schema and explicitly delegated the readiness-signal choice. Suite: 81 PHP + 27 JS = 108 tests (9 new PHP in `IncorrectReviewTest.php`, 7 new JS in `results-message.test.js`, plus 3 new tests for the `csa_scale_exam_duration()` extraction needed because mini-exam counts like 6 or 15 aren't in the preset `[25,50,100,274]` list `csa_plan_exam()` validates against).
**Migration status:** `migration_11.sql` written for production (not yet applied — same as every prior extraction, production wasn't touched this session). A `stg_`-prefixed version + SlickPrepTime package rebuild is still needed before this feature works on staging.
