## 2026-07-21 Hands-on "Show Me How" walkthroughs (Feature 3 of 3-feature plan)

**Symptom:** No feature existed to help users practice mock-exam scenarios hands-on in a real ServiceNow instance — the app only tested recall, not doing.
**Root cause:** N/A — new feature, not a bug.
**Fix:** Added `walkthrough` (TEXT, nullable) to `questions` and `service_now_url` (VARCHAR(255), nullable) to `users` via `migration_10.sql`, folded into the canonical `schema.sql`. Server-side resolution lives in `webapp/lib/walkthrough.php` (`csa_resolve_walkthrough()`, a pure function: per-question walkthrough → category-level template → "coming soon" fallback, then `{{SERVICE_NOW_URL}}` substitution) — the client only ever displays already-resolved text, same split as how `explanation` already worked. 9 tests in `tests/WalkthroughTest.php`. Wired into `questions.php` (Flashcards) and `exam_submit.php`'s post-submission review array — **deliberately not** `exam_start.php`/`exam_active.php` (the live, pre-submission exam-taking endpoints), matching the existing rule that `explanation` never leaks there either; adding a walkthrough to those would hand out answers mid-exam. New `set_service_now_url.php` + `me.php` extension follow the exact existing `set_exam_date.php`/`me.php` pattern. Frontend: "Show Me How" toggle button + collapsible panel in both `flashcards.js` (per-card) and `exam.js` (per-review-row, via one delegated click listener on the stable `reviewList` container rather than per-button listeners, since that container's innerHTML gets fully replaced on every result render). New "ServiceNow Instance" settings card in `account.html`, mirroring the Exam Date card exactly.
**Content status:** 5 of 25 real seeded categories (Service Catalog, CMDB, Forms, Navigation, Security & ACL — chosen for question-count coverage, ~34% of the 279-question bank) have draft category-level templates in `webapp/lib/walkthrough_templates.php`. **These are drafts written from general ServiceNow UI conventions, not verified against a live instance — pending user review before being trusted as accurate.** The other 20 categories fall back to the "coming soon" message rather than getting fabricated content; per-question bespoke walkthroughs (vs. category-level generic ones) were explicitly scoped out as a separate future effort requiring real ServiceNow domain review.
**Why this approach:** Resolution logic in one server-side pure function (not duplicated client-side) keeps it testable the same way as every other extraction in this project. Explicitly did not attempt to auto-generate all 25 categories' worth of "very detailed" content — see the plan discussion earlier this session for why (confident-sounding but wrong step-by-step instructions would actively mislead real exam prep, worse than no walkthrough at all).

**Migration status:** `migration_10.sql` written for production (not yet applied — production wasn't touched this session, same as every prior extraction). A `stg_`-prefixed version needs to be run against the live staging tables via phpMyAdmin before this feature works on SlickPrepTime; see the chat for the exact ALTER statements.

## 2026-07-21 Table-prefix isolation for shared-database staging deployments

**Symptom:** SlickPrepTime (a staging deployment for the next version) was set up assuming a second, isolated InfinityFree database would be available. It wasn't — the account's connection panel confirmed only one database exists (`if0_37565517_csa`), and it's the live production database with real user data. Staging needed genuine data isolation without a second database to isolate into.
**Root cause:** N/A — infrastructure constraint, not a bug. InfinityFree's free tier provides one MySQL database per account.
**Fix:** Added `webapp/lib/table_prefix.php` (a pure function, `csa_prefix_tables(string $sql, string $prefix): string`, plus the explicit `CSA_TABLES` list of all 10 table names) and wired it into `webapp/config/db.php` via a `CsaPrefixedPdo extends PDO` class that overrides `prepare()`/`query()`/`exec()` to rewrite table names before every query. A deployment opts in by setting `table_prefix` in its config array (e.g. `'stg_'`); when unset (production, local dev — neither sets it), `csa_prefix_tables()` is a verified no-op and behavior is completely unchanged.
**Why this approach over editing all 33 API files directly:** Hand-inserting `{$prefix}` into every SQL string across 33 files has 33 files' worth of chances to miss one — and a missed reference would silently read/write live production data, the exact failure mode being defended against. Centralizing the rewrite in the PDO layer means zero API files need to change, and correctness only needs to be proven once. Verified against all 67 real queries extracted from every API file plus `seed.php` (word-boundary regex per known table name) — zero mismatches, zero false positives on columns that share a word-prefix with a table name (`question_id`, `user_id`, `option_text`, etc. are never touched), zero double-prefixing. That verification became `tests/TablePrefixTest.php` (10 tests) rather than staying a throwaway script, matching how every other extraction in this project has been handled. Suite is now 54 PHP + 11 JS = 65 tests.
**Follow-on requirement:** the shared-DB constraint also means `seed.php` must never run against unprefixed tables again (would `TRUNCATE` live data) — restored to the SlickPrepTime package specifically because prefixed queries make it now safe to run there (it'll only ever touch `stg_*` tables through the same PDO wrapper). Also: `api/register.php` has self-registration disabled ("closed group, admin adds accounts"), so there's no code path that creates a `users` row at all — a staging test account has to be inserted directly via phpMyAdmin into `stg_users`, not created through the app.

## 2026-07-18 Fifth test file, first JS tests: extracted Rapid Drill timing (overnight autonomous session)

**Symptom:** No test coverage for Rapid Drill mode's progress-bar tick math or card-index wraparound in `assets/js/flashcards.js`. This is the exact feature the other session manually verified in the browser at the end of the v29 session ("progress bar advances (1.25% → 2.5%)... Pause genuinely freezes it... confirmed the fill % held steady across 1.6s while paused") — a one-time manual check, not a repeatable test.
**Root cause:** N/A — not a bug. The tick-progress calculation and the `(idx + dir + total) % total` wraparound were inline inside a `setInterval` callback and a DOM-event-bound function, both tightly coupled to `document.getElementById(...)` calls.
**Fix:** Extracted `nextDrillIndex()` and `computeDrillTick()` into `webapp/assets/js/lib/drill-timing.js` — a small UMD-style module (works as both a classic `<script>` tag global and a Node `require()`, since this project has no bundler and no ES module usage anywhere). `flashcards.js` now calls into it instead of computing inline; added a `<script>` tag for the new file in `flashcards.html`, loaded before `flashcards.js`. Added `tests/js/drill-timing.test.js` (11 tests) using **Node's built-in `node:test` + `node:assert`** — deliberately zero new npm dependencies for a project with no `package.json` at all. Covers: forward/backward wraparound at both ends of the deck, single-card deck (wraps to itself), first-tick progress amount, a full 80-tick simulation to the first advance (8000ms duration / 100ms tick), the exact-100%-still-advances boundary, and that pausing mid-sequence genuinely freezes progress across multiple ticks (not just one).
**Why this approach:** Same behavior-preservation discipline as the three PHP extractions — ran the original inline logic against the new module on 6 index cases + 5 tick cases + a full 80-tick simulation via a throwaway Node script, zero mismatches, before writing tests. Also discovered along the way: `node --test <directory>` does not work as a bare directory path on this Node version (v24.15.0) — it must be a glob (`tests/js/**/*.test.js`) or explicit file list. Documented in `CLAUDE.md`'s CI Commands so this doesn't get rediscovered. Also confirmed Node's built-in test runner supports `--test-reporter=junit`, so the combined PHP+JS baseline in `.claude/ci-baseline.json` is generated from real structured output for both stacks, not hand-typed for either.

**Scope note:** This covers 5 of 33 API/logic units tested now (4 PHP extractions + 1 JS). The full suite is 49 tests (38 PHP + 11 JS). See `CLAUDE.md`'s "Known Gaps" for what's still untested — this was the third and final endpoint requested for tonight's session; see the morning summary for the full recap and recommendations.

## 2026-07-18 Fourth test file: extracted exam question-count/duration planning (overnight autonomous session)

**Symptom:** No test coverage for `api/exam_start.php`'s question-count validation and proportional timer-scaling formula — the logic that decides how many questions and how much time a Mock Exam gets. This is adjacent to (not the same as) the historical "showed 569 minutes" timer bug DEPLOY.md documents (that one was a UTC/timezone bug, not this formula), but it's the same general risk area: a wrong number here is directly visible to the user as an incorrect countdown.
**Root cause:** N/A — not a bug found. The requested-count validation, the `min(requested, available)` clamp, and the `round(...to nearest 30s...), floor at 60s` duration formula were all inline, split across two separate places in the file (count logic near the top, duration formula near the bottom, with a live DB query for `$allIds` and a `shuffle()` call in between).
**Fix:** Extracted `csa_plan_exam()` into `webapp/lib/exam_planning.php`, consolidating both pieces (previously ~15 lines split across two locations) into one pure function taking the requested count, allowed-counts list, default, total question bank size, and full duration — returning the resolved count and duration together. Added `tests/ExamPlanningTest.php` (9 tests) covering: full-length exam gets full duration, proportional scaling for shorter exams, invalid/zero/negative count falls back to default, the 60-second floor, the round-to-nearest-30-seconds invariant (checked via modulo across all 4 allowed counts), and that the formula isn't hardcoded to the current live 274-question bank size.
**Why this approach:** Same behavior-preservation discipline as the prior extractions — ran the original inline logic against the new function on 11 fixtures (7 initial + 4 targeting different bank sizes) via a throwaway script, zero mismatches, before writing tests. **One real pre-existing defect surfaced and deliberately left unfixed, out of scope for tonight:** the duration formula divides by `$fullTotalQuestions` with no zero-guard — if the question bank were ever empty, this throws `DivisionByZeroError` (PHP 8's `/` operator throws on divide-by-zero, unlike `<= 8.0`). Not reachable today since the bank always has 274 seeded questions, but worth a guard if that ever changes. Documented in the new lib file's docblock and here rather than silently patched, since fixing it wasn't authorized as part of "extract this logic" and changes runtime behavior for a currently-unreachable case.

**Scope note:** This covers 4 of 33 API endpoints now.
