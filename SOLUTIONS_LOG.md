## 2026-07-21 Incorrect Answers Review learning loop (Mock Exam → Flashcards → Mini-Exam)

**Symptom:** No path existed from "you got a question wrong on the mock exam" back to targeted, verified-ready practice — a user who failed or scraped by had to either re-take the full 25-274 question exam or manually hunt down what they'd missed.
**Root cause:** N/A — new feature, not a bug.
**Fix:** Mini-exams are modeled as ordinary `exam_attempts` rows (`migration_11.sql`: `parent_attempt_id`, `attempt_kind ENUM('full','mini')`), reusing 100% of the existing grading/scoring/history code rather than a parallel system — the only new columns needed. `webapp/lib/incorrect_review.php` has two pure functions: `csa_compute_review_readiness()` (known-count / total ≥ 80% threshold) and `csa_pass_percent_for_kind()` (mini-exams need 80% to pass, full exams still 70% — read from `config['exam']['mini_pass_percent']`, falls back to 80 if a deployment's config predates this feature). Readiness signal reuses `flashcard_progress.status = 'known'` (Leitner box ≥ 2, i.e. two successful "Good" reviews) — the exact field that already means "mastered" everywhere else in the app — rather than inventing new tracking or a "Good/Easy" split that doesn't exist in the real flashcard UI (only Again/Good + a separate confidence slider). `exam_start.php` gained a `parentAttemptId` branch: server-derives the wrong-question set from `exam_answers WHERE attempt_id = parentAttemptId AND is_correct = 0` (never a client-supplied ID list, same rule as everywhere else scored data is involved), shuffles, and **caps at 15 questions** even if more were missed — mini-exams stay mini. `questions.php` gained an optional `?attemptId=` filter (same server-side derivation) so Flashcards can show only a specific attempt's misses. New `incorrect_review_status.php` endpoint returns the readiness computation for a given attempt. Frontend: `exam.js`'s results screen branches via a new pure `webapp/assets/js/lib/results-message.js` (`buildResultsMessage()` — headline + CTA struct, no DOM, fully unit tested) into three cases — full exam with misses → "Review Incorrect Answers with Flashcards" link; mini-exam pass → dashboard link; mini-exam fail → back into the same review loop. `flashcards.js` gained an `?mode=incorrect_review&attemptId=X` mode that fetches the filtered set, hides the normal filter/category/search/mode-switch UI (none of it applies to a fixed practice set), and shows a live readiness banner with a "Start Mini-Exam" button once the 80% threshold is hit.
**Design decision — flat, not chained, attempt linkage:** Every mini-exam's `parentAttemptId` is always the root full-exam attempt, never the previous mini-exam. This means (a) the flashcard review pool is always the *original* full set of misses, never narrowing round to round — matching the user's literal "same missed questions" spec — and (b) auditing "all mini-exam attempts for this full exam" is a single flat query instead of a recursive walk. The URL/CTA plumbing was simplified mid-session by dropping a redundant `originalAttemptId` param once this was settled — a single `attemptId` query param suffices everywhere since parent and current collapse to the same value.
**Verified live, not just unit-tested:** Full loop driven end-to-end through a real browser session against a local PHP+MariaDB stack (25-question exam → 6 wrong → flashcards review → 100% known → mini-exam pass with 80% mark shown; separately, a second 25-question exam with 25 wrong → capped to a 15-question mini-exam → deliberate fail → confirmed the "review again" link correctly routes back to the *root* attempt, and that the loop re-offers "Start Mini-Exam" afterward). This caught two real bugs the unit tests didn't: the mini-exam question count was originally unbounded (fixed with the 15-question cap above), and the local dev DB was missing `migration_10` entirely (unrelated pre-existing drift, applied to unblock testing).
**Why this approach:** User's original spec proposed 4 new denormalized columns (`incorrect_question_ids`, `mini_exam_passed`, `mini_exam_attempts`, `flashcards_reviewed`); countered with the 2-column `exam_attempts` extension above since every one of those 4 is either derivable from `exam_answers` (already the source of truth for correctness) or from reusing `exam_attempts`/`flashcard_progress` as real rows — avoiding a second, easily-divergent source of truth for data the app already tracks. User accepted the leaner schema and explicitly delegated the readiness-signal choice. Suite: 81 PHP + 27 JS = 108 tests (9 new PHP in `IncorrectReviewTest.php`, 7 new JS in `results-message.test.js`, plus 3 new tests for the `csa_scale_exam_duration()` extraction needed because mini-exam counts like 6 or 15 aren't in the preset `[25,50,100,274]` list `csa_plan_exam()` validates against).
**Migration status:** `migration_11.sql` written for production (not yet applied — same as every prior extraction, production wasn't touched this session). A `stg_`-prefixed version + SlickPrepTime package rebuild is still needed before this feature works on staging.

## 2026-07-21 Tie walkthroughs to the specific flashcard, not just its category

**Symptom:** User feedback: category-level walkthrough templates read as generic ("here's how Form Layout works") with no reference to the actual question on screen — didn't feel connected to the specific flashcard.
**Root cause:** N/A — design gap, not a bug. The 5 templates never referenced any per-question data.
**Fix:** `csa_resolve_walkthrough()` gained two new required parameters, `$questionContext` and `$correctAnswerContext`, substituted via `{{QUESTION_CONTEXT}}`/`{{CORRECT_ANSWER}}` into whichever text was chosen (per-question or category template) — same substitution mechanism already used for `{{SERVICE_NOW_URL}}`. New `csa_correct_answer_summary(array $options): string` helper builds the answer text ("A: Text" or "A: Text; C: Text" for multi-select) once, shared by both call sites (`questions.php`, `exam_submit.php`) rather than duplicated. All 5 templates rewritten to open with "This flashcard: "..."\nCorrect answer: ..." before the (now more concise, 4-5 step instead of 7-8 step) generic instructions.
**Why this approach, not per-question content:** User's literal ask was "generate walkthrough instructions of each flashcard" — i.e. 279 individually-authored scripts. Explicitly declined to do that: it's the exact accuracy risk flagged before Feature 3 was built (confidently-generated but unverified ServiceNow UI steps, at 55x the surface area of the 5 templates already pending review), and reverses what the user themselves agreed to earlier this session ("I'd rather sanity-check ServiceNow accuracy than trust an AI-generated walkthrough blindly"). Instead, made every resolved walkthrough explicitly reference the specific question and its correct answer using data already trusted (not fabricated), which directly addresses "not related to the flashcard" without inventing new unverified ServiceNow facts per question. 15 tests added/updated in `tests/WalkthroughTest.php` (69 PHP total). Verified against the exact Forms question from the user's own earlier screenshot before considering this done.

## 2026-07-21 Fix: "Show Me How" button appeared to do nothing (nested-button-inside-toggle-card bug)

**Symptom:** Clicking "Show Me How" on a revealed flashcard had no visible effect — reported by the user after deploying Feature 3 to SlickPrepTime.
**Root cause:** `#fcShowMeHowBtn` (and its walkthrough panel) live inside `#flashcard`, which has its own `click` listener that collapses a revealed card back to its question-only state (`if (!revealed) reveal(); else render();`) — the reveal/collapse toggle used for the "click the card again to hide the answer" UX. That listener already excluded clicks inside `#fcNoteBox` but not the new walkthrough button, so a click on "Show Me How" bubbled up, and the card immediately collapsed back, undoing the button's own toggle in the same tick. Same root cause also affects `#flashcard`'s `pointerdown` handler, which arms a swipe-drag gesture (`setPointerCapture`) on any descendant click, including this one.
**Fix:** Added `#fcWalkthroughWrap` to the existing exclusion checks on both `cardEl`'s `pointerdown` and `click` listeners in `flashcards.js`, matching the pattern already used for `#fcNoteBox`. **Also fixed the identical pre-existing bug on `#fcNoteToggle`** ("Add a note") while in there — it had the exact same nested-button flaw and was never excluded either, so opening the note box likely also self-collapsed immediately; this wasn't part of the walkthrough feature but was too directly related (same root cause, same file, one-line fix) to leave broken once found.
**Why this approach:** Diagnosed by fetching the live API response directly (confirmed the server was correctly returning fully-resolved walkthrough text — 1187 characters for the tested Forms question — ruling out any server-side cause before looking at the client) and reading the actual `cardEl` click/pointerdown handlers rather than guessing. Browser-automation click simulation in this session repeatedly fought coordinate/dispatch quirks that obscured rather than clarified the diagnosis past the first clean repro; the fix itself is grounded in direct source reading of the existing, working `#fcNoteBox` exclusion pattern, not in the inconclusive later automation attempts. Asked the user to verify in their real browser rather than trusting a shaky automated repro as final confirmation.

## 2026-07-21 Table-prefix isolation for shared-database staging deployments

**Symptom:** SlickPrepTime (a staging deployment for the next version) was set up assuming a second, isolated InfinityFree database would be available. It wasn't — the account's connection panel confirmed only one database exists (`if0_37565517_csa`), and it's the live production database with real user data. Staging needed genuine data isolation without a second database to isolate into.
**Root cause:** N/A — infrastructure constraint, not a bug. InfinityFree's free tier provides one MySQL database per account.
**Fix:** Added `webapp/lib/table_prefix.php` (a pure function, `csa_prefix_tables(string $sql, string $prefix): string`, plus the explicit `CSA_TABLES` list of all 10 table names) and wired it into `webapp/config/db.php` via a `CsaPrefixedPdo extends PDO` class that overrides `prepare()`/`query()`/`exec()` to rewrite table names before every query. A deployment opts in by setting `table_prefix` in its config array (e.g. `'stg_'`); when unset (production, local dev — neither sets it), `csa_prefix_tables()` is a verified no-op and behavior is completely unchanged.
**Why this approach over editing all 33 API files directly:** Hand-inserting `{$prefix}` into every SQL string across 33 files has 33 files' worth of chances to miss one — and a missed reference would silently read/write live production data, the exact failure mode being defended against. Centralizing the rewrite in the PDO layer means zero API files need to change, and correctness only needs to be proven once. Verified against all 67 real queries extracted from every API file plus `seed.php` (word-boundary regex per known table name) — zero mismatches, zero false positives on columns that share a word-prefix with a table name (`question_id`, `user_id`, `option_text`, etc. are never touched), zero double-prefixing. That verification became `tests/TablePrefixTest.php` (10 tests) rather than staying a throwaway script, matching how every other extraction in this project has been handled. Suite is now 54 PHP + 11 JS = 65 tests.
**Follow-on requirement:** the shared-DB constraint also means `seed.php` must never run against unprefixed tables again (would `TRUNCATE` live data) — restored to the SlickPrepTime package specifically because prefixed queries make it now safe to run there (it'll only ever touch `stg_*` tables through the same PDO wrapper). Also: `api/register.php` has self-registration disabled ("closed group, admin adds accounts"), so there's no code path that creates a `users` row at all — a staging test account has to be inserted directly via phpMyAdmin into `stg_users`, not created through the app.
